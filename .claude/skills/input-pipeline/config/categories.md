# 记忆分类的枚举与定义

本文档定义了记忆分类体系，包括固定基座分类和动态领域分类。

> **注意**：这里的"分类"是业务概念，指 `category_name` 字段的可选值，而非数据库中的独立记录。原子化记忆存储在 Category 表中，通过 `category_name` 字段归属到对应分类。

---

## 1. 分类体系概览

| 类型 | 数量 | 说明 |
|------|------|------|
| 固定基座分类 | 4 个 | 所有用户通用，覆盖核心记忆类型 |
| 动态领域分类 | 2 个 | 根据用户身份生成，覆盖生活/工作主阵地 |

---

## 2. 固定基座分类（4 个）

| 分类名 | 图标 | 定义 | 收录内容 |
|--------|------|------|----------|
| **核心自我** | 👤 | 用户的静态画像、偏好习惯、底层价值观，以及长期的宏观目标 | 个人基本信息、性格特点、行为偏好、人生愿景/年度目标 |
| **情景时间轴** | 🕰️ | 绑定在绝对时间点上的具体事件与微观待办 | 已发生的事件、具体的计划/待办（需携带时间戳） |
| **语义知识库** | 🧠 | 剥离了时间属性的纯粹知识、灵感与资产 | 学习笔记、专业概念解析、阅读感悟、技能心得 |
| **社交关系图谱** | 👥 | 用户与外部世界的人际链接 | 亲友、同事、导师等实体及其背景备注 |

### 情景时间轴的特殊处理

每条记忆需标注状态：

- `happened`：已发生的事件
- `planning`：计划中的待办（可激活提醒触发器）

---

## 3. 动态领域分类（2 个）

### 3.1 定义

系统冷启动时，LLM 根据用户的身份画像，一次性生成的"生活/工作的核心主阵地"。

### 3.2 设计要求

- 颗粒度要大，足以容纳该身份下高频发生的长尾事项
- 与 4 个固定分类互斥，不重叠
- 覆盖用户的核心生活/工作场景

### 3.3 优秀示例

| 身份 | 动态分类 1 | 动态分类 2 |
|------|-----------|-----------|
| 学生 | [学校生活] 社团、选课、考勤等 | [考试与升学] 备考资料、分数追踪 |
| 教师 | [教学教务] 教学计划、备课进度 | [学生管理] 班级琐事、家长沟通 |
| 程序员/研究员 | [项目研发] 代码架构、排错记录 | [职业发展] 面试准备、技术栈规划 |
| 自由职业者 | [项目作品] 项目进度、客户需求 | [个人品牌] 社交媒体、作品集 |

### 3.4 生成逻辑

```
用户画像数据
      ↓
┌─────────────────────────────────────────┐
│ 固定基座分类名称（4 个）                   │
│ - 核心自我、情景时间轴                     │
│ - 语义知识库、社交关系图谱                 │
└─────────────────────────────────────────┘
      ↓
┌─────────────────────────────────────────┐
│ 动态领域分类名称（2 个）                   │
│ - LLM 根据用户身份生成                    │
│ - 颗粒度大，覆盖生活/工作主阵地            │
└─────────────────────────────────────────┘
      ↓
┌─────────────────────────────────────────┐
│ 用户初始化信息作为原子化记忆               │
│ - 存入 Category 表                       │
│ - category_name 设为对应分类名            │
└─────────────────────────────────────────┘
```

### 3.5 LLM Prompt 示例

```
你是一个记忆分类专家。请根据用户画像，生成 2 个动态领域分类。

用户画像：
- 身份：高中生
- 学段：高二文科
- 兴趣：文学、电影、心理学

已有固定分类：
1. 核心自我：个人画像、偏好习惯、长期目标
2. 情景时间轴：绑定时间的事件和待办
3. 语义知识库：纯粹的知识和灵感
4. 社交关系图谱：人际链接

要求：
1. 生成的分类要与固定分类互斥
2. 颗粒度要大，覆盖该身份下高频发生的长尾事项
3. 分类名简洁有力，描述清晰
4. 优先选择具有长期积累价值、且能容纳至少 50 条以上不同信息的领域
5. 避免过于宽泛（如"日常生活"）或过于狭窄（如"吃早饭"）的分类
6. 返回 JSON 格式：[{"name": "分类名", "description": "分类描述"}]
```

> **分类收敛原则**：LLM 有时会生成过于宽泛或过于狭窄的分类。应在 Prompt 中明确：
> - 分类需具备**长期积累价值**
> - 每个分类应能容纳 **50 条以上**不同类型的信息

### 3.6 LLM 返回示例

```json
[
  {"name": "学校生活", "description": "社团活动、选课考勤、校园琐事等学校日常"},
  {"name": "考试与升学", "description": "备考资料、分数追踪、升学规划等学业进度"}
]
```

---

## 4. 不同分类的去重阈值

| 分类 | 去重阈值 | 原因 |
|------|---------|------|
| **核心自我** | 0.85 | 身份信息重复存储价值低 |
| **情景时间轴** | 0.90 | 时间点不同即视为不同事件 |
| **语义知识库** | 0.88 | 知识点可从不同角度记录 |
| **社交关系图谱** | 0.85 | 同一人信息应合并 |
| **动态领域分类** | 0.92 | 项目进度等需保留演进记录 |

---

## 5. 固定分类保护规则

以下分类**不参与遗忘机制**，只增不改：

- **核心自我**：身份信息、偏好习惯是用户的核心特征，不应遗忘
- **社交关系图谱**：人际关系是长期记忆，不应遗忘
