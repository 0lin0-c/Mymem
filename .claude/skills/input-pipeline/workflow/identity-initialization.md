# 用户初始化与模板生成

本文档定义了用户首次进入时的初始化流程，包括画像提取、模板生成和分类初始化。

> **前端表单字段设计**：详见 `frontend-design` skill

---

## 1. 整体流程

```
前端提交初始化数据
      ↓
┌─────────────────────────────────────────────────────┐
│ 后端处理流程                                         │
│   ├─ 解析请求数据                                    │
│   ├─ 将 username 存入 User 表                       │
│   ├─ 生成精简版 user_prompt_template，存入 User 表  │
│   ├─ 生成 agent_persona_template，存入 User 表      │
│   ├─ 动态分类生成（根据 LLM 可用性）：                │
│   │   ├─ LLM 可用 → 调用 LLM 生成 2 个个性化分类    │
│   │   └─ LLM 不可用 → 使用占位符，延后生成          │
│   └─ 将用户初始化信息存入 Category 表               │
└─────────────────────────────────────────────────────┘
      ↓
初始化完成，返回响应
```

---

## 2. 动态分类生成策略

### 2.1 混合方案

根据 LLM 可用性决定生成策略：

| LLM 状态 | 生成策略 | 动态分类名称 |
|----------|----------|--------------|
| 可用（服务端配置或用户已配置） | 立即生成 | 个性化分类（如"学校生活"、"考试与升学"） |
| 不可用 | 占位符 | `"专属分类1（待配置）"`、`"专属分类2（待配置）"` |

### 2.2 LLM 可用性判断

```python
def _is_llm_available(self) -> bool:
    """判断 LLM 是否可用

    判断逻辑：
    1. 服务端全局配置：检查 Settings 中是否配置了 API Key
    2. 用户级配置：检查用户是否设置了自己的 API Key（未来扩展）
    """
    from core.config import settings

    # 服务端全局 LLM 配置
    if settings.OPENAI_API_KEY or settings.ANTHROPIC_API_KEY:
        return True

    # TODO: 用户级 LLM 配置（未来扩展）
    # if user.llm_config and user.llm_config.api_key:
    #     return True

    return False
```

### 2.3 占位符分类后续更新

当用户配置 LLM 后，触发动态分类生成：

```
用户配置 LLM API Key
      ↓
检查是否使用占位符分类
      ↓
是 → 调用 LLM 生成个性化分类 → 更新 Category 表
否 → 跳过
```

### 2.4 默认分类映射

当 LLM 不可用时，可根据用户身份类型使用预设默认分类：

| 身份类型 | 动态分类 1 | 动态分类 2 |
|----------|-----------|-----------|
| 学生 | 学校生活 | 考试与升学 |
| 上班族 | 项目研发 | 职业发展 |
| 教师 | 教学教务 | 学生管理 |
| 自由职业者 | 项目作品 | 个人品牌 |
| 其他 | 生活记录 | 兴趣探索 |

## 3. 模板生成逻辑

### 3.1 内容平衡策略

**核心问题**：`user_prompt_template`（每次加载）和 details 分类中的"核心自我"如何分工？

**解决方案：分层存储**

```
┌─────────────────────────────────────────────────────────┐
│ user_prompt_template（每次对话加载，~100字）             │
│ - 基本信息：姓名、身份类型、关键属性                      │
│ - 被反复提及的偏好：对话中多次出现的偏好会被提取到这里     │
│ - 用户着重强调的爱好：用户明确表示重要的兴趣             │
│ → AI 每次对话都能知道"用户是谁"和"用户在意什么"          │
└─────────────────────────────────────────────────────────┘
                          ↓ 检索时按需加载
┌─────────────────────────────────────────────────────────┐
│ "核心自我"分类下的 Resources                             │
│ - 完整的身份背景、偏好习惯、长期目标                     │
│ - 作为 Resource 存储，可被检索                           │
└─────────────────────────────────────────────────────────┘
```

**动态更新机制**：

- 初始化时：`user_prompt_template` 只包含基本信息
- 对话过程中：LLM 发现用户反复提及某偏好，或用户强调某爱好 → 自动补充到 `user_prompt_template`
- 完整信息始终存储在"核心自我"分类下

---

## 4. user_prompt_template

### 4.1 生成规则

基本信息 + 核心偏好/强调的爱好，控制在 100 字以内。

### 4.2 初始模板示例

```
用户小明，高中生，高二文科方向。
```

### 4.3 对话中动态补充后的示例

```
用户小明，高中生，高二文科方向。
偏好：晚上学习效率更高，喜欢通过例子理解概念。
重点兴趣：对心理学特别感兴趣，希望深入了解认知心理学。
```

---

## 5. agent_persona_template（AI 人设模板）

### 5.1 生成规则

将 AI 的名字、角色、性格、沟通风格整合为行为指令，控制在 100 字以内。

### 5.2 模板示例

```
你是小智，用户的学习导师。
性格：温柔耐心、严谨专业。
称呼用户"小明"，自称"小智"。
沟通风格：日常轻松，多用鼓励式交流，循循善诱。
```

## 7. System Prompt 加载机制

每次对话时，系统自动拼接两个模板：

```python
# 构建 System Prompt（~250字）
system_prompt = ""
if user.user_prompt_template:
    system_prompt += user.user_prompt_template.decode('utf-8')
if user.agent_persona_template:
    system_prompt += "\n\n" + user.agent_persona_template.decode('utf-8')
```

**最终发送给 LLM 的内容**：

```
用户小明，高中生，高二文科方向。

你是小智，用户的学习导师。
性格：温柔耐心、严谨专业。
称呼用户"小明"，自称"小智"。
沟通风格：日常轻松，多用鼓励式交流，循循善诱。
```

**需要详细信息时**：通过检索"核心自我"分类获取完整内容。
