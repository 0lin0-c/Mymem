# LLM 充足性判断

本文档定义检索流程的第三步：判断 Category 层检索结果是否已经足够回答用户问题。

---

## 1. 设计目的

避免不必要的 Resource 层检索，同时不能因为“过早停下”而错过更精确的证据。

| 场景 | 判断结果 | 后续动作 |
|------|----------|----------|
| Category 结果已经明确给出回答所需的关键信息 | 足够 | 直接构建上下文回答 |
| Category 结果语义相关但缺少关键细节 | 不足 | 继续进入 Resource 层检索 |
| 没有 Category 结果 | 不足 | 直接进入 Resource 层检索 |

---

## 2. Prompt 输出格式

让 LLM 基于以下信息做二元判断：

- 用户问题
- 检索到的 Category 记忆片段

输出 JSON：

```json
{
  "sufficient": true,
  "reason": "简要说明判断原因"
}
```

---

## 3. 判断规则

1. 只有当记忆片段已经显式覆盖回答问题所需的全部关键事实时，才判定为“足够”。
2. 如果片段语义相关，但缺少日期、人名、地点、数量等精确信息，判定为“不足”。
3. 对 `when / who / where / how many` 这类精确问法，必须出现可直接作答的事实，不能靠模糊相关性放行。
4. 若存在合理疑点，优先判定为“不足”，继续走 Resource fallback。

---

## 4. 降级策略

当 LLM 充足性判断失败时：

- 若 Category 层存在单条高分结果，可判定为“足够”。
- 不再使用“结果条数多就视为足够”的规则，避免 `category_only` 提前截断。
- 其他情况一律进入 Resource 层继续检索。
