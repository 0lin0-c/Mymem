# 运行安全规则

本文档定义本项目在连接真实数据库表进行评测、检索、回答、导入和重建记忆时的运行安全边界。

本文档的目标不是强制把所有测试迁移到独立测试库，而是明确以下原则：

- 允许直接使用真实数据库表。
- 默认应当是只读模式。
- 只有在用户明确要求“重新进行记忆储存 / reset / reimport”时，才允许进入写入或清理模式。
- 测试框架不得绕过业务意图，自动对真实数据库执行清表、删库、重建等破坏性操作。

---

## 1. 总原则

本项目默认采用“真实库优先，但权限分层”的运行规则。

具体含义如下：

- 可以直接连接当前正在使用的真实数据库表进行评测。
- 如果当前任务只是检索、回答、分析、复盘、bad-case 诊断，则必须视为只读任务。
- 只读任务不得修改真实库中的 `users`、`resources`、`categories`、`resource_categories` 等数据。
- 只有在用户明确授权重新导入、重新建记忆、重置记忆时，才允许进入写入任务。
- 清理旧记忆、覆盖旧记忆、删除旧数据都属于高风险写操作，必须由显式参数或明确用户指令触发。

一句话概括：

**真实库可以直接用，但默认只能读；要写、要删、要重建，必须显式授权。**

---

## 2. 名词定义

### 2.1 真实数据库

指当前 `.env` / `settings.database_url` 所连接的实际业务数据库，可能已经包含用户真实评测数据、已导入记忆和历史检索状态。

### 2.2 只读任务

指不会改变记忆存储状态的任务，例如：

- 仅检索已有记忆
- 基于已有记忆生成回答
- assistant_eval 的 retrieval-only 模式
- retrieval_eval 的 retrieval-only 模式
- bad-case diagnosis
- SQL trace 对比分析
- 读取历史结果文件并做报告分析

### 2.3 写入任务

指会改变数据库状态的任务，例如：

- 导入 converted_data
- 重新进行记忆存储
- reset 某个测试用户的旧记忆后重建
- onboarding 新测试用户
- 任何 `INSERT` / `UPDATE` / `DELETE`

### 2.4 高风险破坏性操作

包括但不限于：

- 清空整库表数据
- 清空多个用户数据
- 删除某个测试用户的全部旧记忆
- fixture teardown 时自动 `DELETE`
- 自动重建后覆盖旧状态

这些操作默认禁止，只有显式授权时才允许。

---

## 3. 运行模式

### 3.1 只读评测模式

适用场景：

- `retrieval_only`
- 只跑 retrieval_eval
- 只跑 assistant_eval 但使用已有记忆
- 检索结果复盘
- bad-case 分析
- SQL 排名分析

权限规则：

- 允许连接真实数据库
- 允许读取所有相关表
- 禁止导入数据
- 禁止 reset 旧记忆
- 禁止任何 teardown cleanup
- 禁止任何自动清表

这个模式应当成为大多数评测和分析任务的默认模式。

### 3.2 写入评测模式

适用场景：

- 用户明确要求重新进行记忆储存
- 用户明确要求重新导入 converted_data
- 用户明确要求 reset/reimport 某个测试用户

权限规则：

- 允许连接真实数据库
- 允许导入记忆
- 如确有需要，允许只删除“当前被明确指定的测试用户”的旧记忆
- 不允许扩大到全局清表
- 不允许隐式触发破坏性 cleanup

这个模式下的任何重置行为都必须是显式、可见、可追踪的。

### 3.3 开发单测模式

适用场景：

- unit test
- contract test
- 纯逻辑验证

这类模式不应默认直连真实库执行清理逻辑。

如果确实要跑这类测试，应满足至少一个条件：

- 使用独立测试数据库
- 或明确进入不会清理真实库的只读模式
- 或完全使用 mock / fake / isolated fixture

---

## 4. `pytest` 与 `retrieval_only` 的关系

`pytest` 和 `retrieval_only` 不是同一层概念。

### 4.1 `pytest`

`pytest` 是外层测试执行框架，负责：

- 启动测试
- 加载 `tests/conftest.py`
- 创建 fixture
- 管理 setup / teardown 生命周期

### 4.2 `retrieval_only`

`retrieval_only` 是内层评测参数，负责表达：

- 本次评测跳过重新导入
- 直接基于已有数据库记忆做检索和回答

### 4.3 关键结论

即使某次评测业务逻辑上是 `retrieval_only`，只要它仍然通过 `pytest` 运行，并且 `pytest` fixture 会在 teardown 阶段清表，那么最终仍然可能对真实库造成破坏。

因此：

- `retrieval_only` 只能约束业务评测逻辑
- 不能自动约束 `pytest` fixture 的 teardown 行为

这也是本次事故的根本原因之一。

---

## 5. 当前已知风险

当前 `tests/conftest.py` 中的数据库 fixture 采用如下模式：

- 使用 `settings.database_url` 创建测试引擎
- 在 session 结束时执行清表 teardown

如果 `settings.database_url` 直接指向真实库，那么任何进入该 fixture 生命周期的 `pytest` 测试，都可能在测试结束时清空真实库表数据。

这与本文档的安全规则直接冲突。

需要强调：

- 问题不在于 retrieval-only 业务逻辑本身会主动删库
- 问题在于测试框架 teardown 覆盖了业务的“只读意图”

---

## 6. 强制规则

后续所有测试与评测命令必须遵守以下规则。

### 规则 A：真实库默认只读

只要任务目标是“检索 / 回答 / 分析”，就必须进入只读模式。

这类任务不得触发：

- `DELETE`
- `INSERT`
- `UPDATE`
- teardown cleanup
- reset_memory
- 自动导入

### 规则 B：只有显式授权才允许重建记忆

只有当用户明确提出以下意图之一时，才允许进入写入模式：

- 重新进行记忆储存
- 重新导入 converted_data
- reset 某个测试用户
- 重跑完整入库流程

如果没有显式授权，则默认禁止任何写操作。

### 规则 C：禁止测试框架隐式清库

无论是 unit test、contract test、eval runner 还是辅助分析脚本：

- 不得在真实库上执行自动 teardown 清表
- 不得在任务结束时做隐式 cleanup
- 不得因为“测试跑完了”就恢复为空库

### 规则 D：破坏性操作必须范围受限

即使用户明确要求重建记忆，也只能影响本次明确指定的测试用户或明确指定的数据集范围。

禁止：

- 清空全库
- 删除无关用户数据
- 通过广义 fixture cleanup 影响整个数据库

### 规则 E：命令行为必须与用户意图一致

如果用户要求：

- “只跑 retrieval-only”
- “只分析”
- “只看检索和回答”

那么系统必须保证整个执行链路都是只读的，而不是仅业务层只读、测试框架层仍可删库。

## 7. 推荐运行矩阵

### 7.1 只读命令

典型场景：

- retrieval_only assistant_eval
- retrieval_only retrieval_eval
- bad-case diagnosis
- SQL 分析

要求：

- 允许读取真实库
- 禁止 reset
- 禁止导入
- 禁止 cleanup

### 7.2 写入命令

典型场景：

- 重新导入 sample
- 重新建记忆
- reset 某个测试用户

要求：

- 必须有显式授权
- 必须明确写明影响范围
- 必须避免扩大到全库

### 7.3 普通 pytest 单测

典型场景：

- `tests/test_services/...`
- `tests/unit/...`
- `tests/contract/...`

要求：

- 不得默认直接连接真实库并自动清理
- 若连接真实库，只能只读
- 若需要清理和回滚能力，应使用独立测试数据库或隔离 fixture

---

## 8. 实施建议

为了让本文档真正落地，后续代码实现至少应覆盖以下保护：

1. 在 `tests/conftest.py` 中增加真实库保护
   - 如果当前数据库 URL 指向真实库且 fixture 包含 destructive cleanup，则直接拒绝运行

2. 将 converted_data 评测命令明确区分为只读模式和写入模式
   - `retrieval_only` 强制只读
   - `reset_memory` 强制显式写入模式

3. 将 teardown cleanup 从真实库路径中移除
   - 不允许对真实库执行自动清表

4. 在命令日志中显式打印本次运行模式
   - `READ_ONLY_REAL_DB`
   - `WRITE_REAL_DB_EXPLICIT_RESET`
   - `ISOLATED_TEST_DB`

这样用户在执行前就能明确知道本次命令是否会改库。

---

## 9. 当前项目的执行基准

在当前项目语境下，默认应采用以下执行基准：

- 评测已有记忆：真实库 + 只读
- 检索复盘：真实库 + 只读
- 回答复盘：真实库 + 只读
- 重新导入记忆：真实库 + 显式授权写入
- 普通 pytest 验证：不得再默认对真实库执行 teardown 清理

---

## 10. 一句话规范

**本项目允许直接使用真实数据库表进行评测，但默认只能读；只有用户明确要求重新进行记忆储存时，才允许写入、重置或清理。**
