# 💬 Chat 对话保存流程设计

## 1. 整体流程

```
用户输入 → 用户识别 → 路由分发（按 modality） → 对应 Handler → LLM 分析 → 保存 Resource → 返回
```

---

## 2. 用户识别机制

### 2.1 识别流程

```
用户发送请求
      ↓
系统检查是否已识别该会话的用户
      ↓
┌─────────────────────────────────────────────────────┐
│ 未识别：LLM 询问"请问你是谁？"                        │
│         ↓                                           │
│ 用户回答（如"我是张三"）                              │
│         ↓                                           │
│ LLM 解析用户身份 → 匹配/创建用户 → 记录 user_id       │
└─────────────────────────────────────────────────────┘
      ↓
已识别：直接使用 user_id 处理请求
```

### 2.2 实现阶段

| 阶段 | 识别方式 | 说明 |
|------|---------|------|
| **当前（无前端）** | LLM 对话识别 | 每次新会话，LLM 主动询问用户身份 |
| **后续（有前端）** | 登录态自动识别 | 从 JWT Token / Session 获取 user_id |

---

## 3. 身份初始化流程

新用户首次进入系统时，需要完成双向身份初始化：
1. **用户画像**：让系统了解用户是谁
2. **AI 助手定制**：让用户定义自己的专属 AI 助手

### 3.1 整体流程

```
新用户首次进入
      ↓
┌─────────────────────────────────────────────────────┐
│ 第一阶段：用户画像初始化                              │
│   ├─ 基本信息（用户名、身份类型）                     │
│   ├─ 身份详情（根据类型动态展示）                     │
│   └─ 使用场景 & 兴趣标签（可选）                      │
└─────────────────────────────────────────────────────┘
      ↓
┌─────────────────────────────────────────────────────┐
│ 第二阶段：AI 助手定制                                │
│   ├─ AI 名字（用户给助手起名）                        │
│   ├─ AI 身份角色                                     │
│   ├─ AI 性格特点                                     │
│   └─ 沟通风格                                        │
└─────────────────────────────────────────────────────┘
      ↓
┌─────────────────────────────────────────────────────┐
│ 第三阶段：后端自动处理                               │
│   ├─ 生成精简版 user_prompt_template（基本信息）      │
│   ├─ 生成 agent_persona_template（AI 人设）          │
│   ├─ 生成 2 个动态分类名称                           │
│   ├─ 将用户初始化信息作为原子化记忆存入 Category 表   │
│   └─ 存入数据库                                      │
└─────────────────────────────────────────────────────┘
      ↓
初始化完成，开始对话
```

### 3.2 用户画像字段设计

> **说明**：以下字段为前端初始化流程需要收集的信息，后端处理后统一整合存入数据库，不单独存为字段。

> - 其余信息整合后存入 `user_prompt_template` 字段

### 3.3 AI 助手定制字段设计

> **说明**：以下字段为前端初始化流程需要收集的信息，后端处理后统一整合存入 `agent_persona_template` 字段。

#### 3.3.1 基础定制

| 字段 | 类型 | 说明 | 选项 |
|------|------|------|------|
| `ai_name` | string | AI 助手的名字 | 用户自定义，如"小助手"、"Jarvis" |
| `ai_role` | enum | AI 的身份定位 | 见下表 |

**AI 身份角色**：

| 角色 | 说明 | 适用场景 |
|------|------|---------|
| 助手 | 高效可靠的执行者 | 日常任务、信息查询 |
| 导师 | 循循善诱的引导者 | 学习辅导、技能提升 |
| 朋友 | 平等亲近的伙伴 | 情感陪伴、日常聊天 |
| 顾问 | 专业理性的分析师 | 决策建议、问题诊断 |
| 伙伴 | 共同成长的合作者 | 创意协作、项目推进 |

#### 3.3.2 性格特点（单选或多选）

| 性格 | 说明 | 典型表现 |
|------|------|---------|
| 严谨专业 | 逻辑清晰、表述精确 | 使用专业术语，提供数据支撑 |
| 幽默风趣 | 轻松活泼、善用比喻 | 适当开玩笑，氛围轻松 |
| 温柔耐心 | 细致周到、不厌其烦 | 多角度解释，鼓励式交流 |
| 简洁高效 | 直奔主题、言简意赅 | 减少客套，快速给答案 |
| 热情活泼 | 积极主动、充满能量 | 主动提问，表达兴趣 |

#### 3.3.3 沟通风格

| 字段 | 类型 | 选项 |
|------|------|------|
| `communication_style` | enum | 正式 / 随意 / 学术 / 日常 |

> **称呼约定**：AI 称呼用户时直接使用用户的 `username`，AI 自称使用 `ai_name`，无需额外配置。

### 3.4 模板生成逻辑

#### 3.4.1 内容平衡策略

**核心问题**：`user_prompt_template`（每次加载）和 details 分类中的"核心自我"如何分工？

**解决方案：分层存储**

```
┌─────────────────────────────────────────────────────────┐
│ user_prompt_template（每次对话加载，~100字）             │
│ - 基本信息：姓名、身份类型、关键属性                      │
│ - 被反复提及的偏好：对话中多次出现的偏好会被提取到这里     │
│ - 用户着重强调的爱好：用户明确表示重要的兴趣             │
│ → AI 每次对话都能知道"用户是谁"和"用户在意什么"          │
└─────────────────────────────────────────────────────────┘
                          ↓ 检索时按需加载
┌─────────────────────────────────────────────────────────┐
│ "核心自我"分类下的 Resources                             │
│ - 完整的身份背景、偏好习惯、长期目标                     │
│ - 作为 Resource 存储，可被检索                           │
└─────────────────────────────────────────────────────────┘
```

**动态更新机制**：
- 初始化时：`user_prompt_template` 只包含基本信息
- 对话过程中：LLM 发现用户反复提及某偏好，或用户强调某爱好 → 自动补充到 `user_prompt_template`
- 完整信息始终存储在"核心自我"分类下

#### 3.4.2 user_prompt_template

**生成规则**：基本信息 + 核心偏好/强调的爱好，控制在 100 字以内。

**初始模板示例**：
```
用户小明，高中生，高二文科方向。
```

**对话中动态补充后的示例**：
```
用户小明，高中生，高二文科方向。
偏好：晚上学习效率更高，喜欢通过例子理解概念。
重点兴趣：对心理学特别感兴趣，希望深入了解认知心理学。
```

#### 3.4.3 agent_persona_template（AI 人设模板）

**生成规则**：将 AI 的名字、角色、性格、沟通风格整合为行为指令，控制在 100 字以内。

**模板示例**：
```
你是小智，用户的学习导师。
性格：温柔耐心、严谨专业。
称呼用户"小明"，自称"小智"。
沟通风格：日常轻松，多用鼓励式交流，循循善诱。
```

#### 3.4.4 分类中的详细存储

初始化时，将完整的用户画像存入"核心自我"分类：

**"核心自我"分类下的 Resource**：
```
【身份信息】
姓名：小明
身份类型：学生
学段：高二
方向：文科
学校：XX中学（可选）

【兴趣领域】
文学：喜欢现代文学、散文
电影：悬疑、科幻类型
心理学：对认知心理学感兴趣

【偏好习惯】
- 偏好晚上学习
- 喜欢通过例子理解概念
- 希望得到鼓励式反馈

【长期目标】
- 提高语文成绩
- 考上理想大学
- 提升写作能力
```

#### 3.4.5 System Prompt 加载机制

每次对话时，系统自动拼接两个模板：

```python
# 构建 System Prompt（~250字）
system_prompt = ""
if user.user_prompt_template:
    system_prompt += user.user_prompt_template.decode('utf-8')
if user.agent_persona_template:
    system_prompt += "\n\n" + user.agent_persona_template.decode('utf-8')
```

**最终发送给 LLM 的内容**：
```
用户小明，高中生，高二文科方向。

你是小智，用户的学习导师。
性格：温柔耐心、严谨专业。
称呼用户"小明"，自称"小智"。
沟通风格：日常轻松，多用鼓励式交流，循循善诱。
```

**需要详细信息时**：通过检索"核心自我"分类获取完整内容。

### 3.5 数据存储设计

User 表已有两个模板字段用于存储个性化配置：

| 字段 | 类型 | 说明 |
|------|------|------|
| `user_prompt_template` | LargeBinary | 用户的全局客观画像（替代 USER.md） |
| `agent_persona_template` | LargeBinary | 助手的性格、语气指令（替代 SOUL.md） |

### 3.6 初始分类生成

#### 3.6.1 设计思路

**核心问题**：分类太多会导致 LLM 分类困难，边界模糊。

**解决方案**：精简为 4+2 分类体系

| 类型 | 数量 | 说明 |
|------|------|------|
| 固定基座分类 | 4 个 | 所有用户通用，覆盖核心记忆类型 |
| 动态领域分类 | 2 个 | 根据用户身份生成，覆盖生活/工作主阵地 |

> **注意**：这里的"分类"是业务概念，指 `category_name` 字段的可选值，而非数据库中的独立记录。原子化记忆存储在 Category 表中，通过 `category_name` 字段归属到对应分类。

#### 3.6.2 固定基座分类（4 个）

| 分类名 | 图标 | 定义 | 收录内容 |
|--------|------|------|----------|
| **核心自我** | 👤 | 用户的静态画像、偏好习惯、底层价值观，以及长期的宏观目标 | 个人基本信息、性格特点、行为偏好、人生愿景/年度目标 |
| **情景时间轴** | 🕰️ | 绑定在绝对时间点上的具体事件与微观待办 | 已发生的事件、具体的计划/待办（需携带时间戳） |
| **语义知识库** | 🧠 | 剥离了时间属性的纯粹知识、灵感与资产 | 学习笔记、专业概念解析、阅读感悟、技能心得 |
| **社交关系图谱** | 👥 | 用户与外部世界的人际链接 | 亲友、同事、导师等实体及其背景备注 |

**情景时间轴的特殊处理**：

每条记忆需标注状态：
- `happened`：已发生的事件
- `planning`：计划中的待办（可激活提醒触发器）

#### 3.6.3 固定分类示例

| 分类名 | 示例记忆 |
|--------|----------|
| **核心自我** | "我是做深度学习研究的"、"我不喜欢吃香菜"、"我今年的目标是发表一篇顶会论文" |
| **情景时间轴** | "我刚刚从健身房练完腿回来"（happened）、"明早十点提醒我提交项目代码"（planning） |
| **语义知识库** | "GNN 结合 Transformer 求解 PDE 的核心优势在于..." |
| **社交关系图谱** | "导师王教授对论文格式的要求非常严格" |

#### 3.6.4 动态领域分类（2 个）

**定义**：系统冷启动时，LLM 根据用户的身份画像，一次性生成的"生活/工作的核心主阵地"。

**设计要求**：
- 颗粒度要大，足以容纳该身份下高频发生的长尾事项
- 与 4 个固定分类互斥，不重叠
- 覆盖用户的核心生活/工作场景

**优秀示例**：

| 身份 | 动态分类 1 | 动态分类 2 |
|------|-----------|-----------|
| 学生 | [学校生活] 社团、选课、考勤等 | [考试与升学] 备考资料、分数追踪 |
| 教师 | [教学教务] 教学计划、备课进度 | [学生管理] 班级琐事、家长沟通 |
| 程序员/研究员 | [项目研发] 代码架构、排错记录 | [职业发展] 面试准备、技术栈规划 |
| 自由职业者 | [项目作品] 项目进度、客户需求 | [个人品牌] 社交媒体、作品集 |

**反例（颗粒度太细，不推荐）**：
- ❌ "文学阅读"、"电影观影" → 太细，应归入语义知识库或合并
- ❌ "学科学习"、"考试复习" → 与学校生活/考试升学重叠

#### 3.6.5 生成逻辑

```
用户画像数据
      ↓
┌─────────────────────────────────────────┐
│ 固定基座分类名称（4 个）                   │
│ - 核心自我、情景时间轴                     │
│ - 语义知识库、社交关系图谱                 │
└─────────────────────────────────────────┘
      ↓
┌─────────────────────────────────────────┐
│ 动态领域分类名称（2 个）                   │
│ - LLM 根据用户身份生成                    │
│ - 颗粒度大，覆盖生活/工作主阵地            │
└─────────────────────────────────────────┘
      ↓
┌─────────────────────────────────────────┐
│ 用户初始化信息作为原子化记忆               │
│ - 存入 Category 表                       │
│ - category_name 设为对应分类名            │
└─────────────────────────────────────────┘
```

**LLM Prompt 示例**（生成动态领域分类）：
```
你是一个记忆分类专家。请根据用户画像，生成 2 个动态领域分类。

用户画像：
- 身份：高中生
- 学段：高二文科
- 兴趣：文学、电影、心理学

已有固定分类：
1. 核心自我：个人画像、偏好习惯、长期目标
2. 情景时间轴：绑定时间的事件和待办
3. 语义知识库：纯粹的知识和灵感
4. 社交关系图谱：人际链接

要求：
1. 生成的分类要与固定分类互斥
2. 颗粒度要大，覆盖该身份下高频发生的长尾事项
3. 分类名简洁有力，描述清晰
4. 返回 JSON 格式：[{"name": "分类名", "description": "分类描述"}]
```

**LLM 返回示例**：
```json
[
  {"name": "学校生活", "description": "社团活动、选课考勤、校园琐事等学校日常"},
  {"name": "考试与升学", "description": "备考资料、分数追踪、升学规划等学业进度"}
]
```

#### 3.6.6 分类稳定性说明

**固定基座分类**：
- 所有用户通用，确保统一的基础框架
- 覆盖四种核心记忆类型：自我、时间、知识、社交
- LLM 分类时优先匹配这 4 个分类

**动态领域分类**：
- 根据用户身份定制
- 对话中可新建更多细粒度分类名称

### 3.7 API 设计

#### 3.7.1 POST /v1/user/onboarding

初始化用户画像和 AI 定制（首次进入时调用）。

**请求**：
```json
{
  "username": "小明",
  "identity_type": "student",
  "identity_detail": {
    "education_stage": "high_school",
    "major": null
  },
  "use_cases": ["学习辅助", "创意灵感"],
  "interests": ["文学", "电影"],
  "ai_customization": {
    "ai_name": "小智",
    "ai_role": "导师",
    "personality": ["温柔耐心", "严谨专业"],
    "communication_style": "日常"
  }
}
```

**响应**：
```json
{
  "success": true,
  "user_id": "uuid-xxx",
  "user_prompt_template": "用户小明，高中生，高二文科方向。",
  "agent_persona_template": "你是小智，用户的学习导师。性格：温柔耐心、严谨专业。称呼用户"小明"，自称"小智"。",
  "initial_categories": {
    "fixed": [
      {"name": "核心自我", "description": "个人画像、偏好习惯、长期目标", "is_fixed": true},
      {"name": "情景时间轴", "description": "绑定时间的事件和待办", "is_fixed": true},
      {"name": "语义知识库", "description": "纯粹的知识、灵感与资产", "is_fixed": true},
      {"name": "社交关系图谱", "description": "人际链接与关系备注", "is_fixed": true}
    ],
    "dynamic": [
      {"name": "学校生活", "description": "社团活动、选课考勤、校园琐事等学校日常", "is_fixed": false},
      {"name": "考试与升学", "description": "备考资料、分数追踪、升学规划等学业进度", "is_fixed": false}
    ]
  },
  "message": "身份初始化完成，可以开始对话了"
}
```

> **注意**：`is_fixed` 字段仅用于前端显示，不存储在数据库中。分类名称作为 `category_name` 的可选值使用。
```

**处理逻辑**：
1. 解析请求数据
2. 将 `username` 存入 User 表
3. 生成精简版 `user_prompt_template`（基本信息），存入 User 表
4. 生成 `agent_persona_template`，存入 User 表
5. 调用 LLM 生成 2 个动态分类名称
6. 将用户初始化信息作为原子化记忆存入 Category 表（`category_name` 设为"核心自我"等）

#### 3.7.2 PUT /v1/user/profile

## 4. 多轮对话缓存机制

### 4.1 设计目标

- **chat 模态**：每 5 轮批量存一次数据库（减少 API 调用和数据库写入）
- **其他模态**（image/video/voice/document）：每次立即存数据库
- **跨模态保护**：chat 计数不受其他模态影响，继续累积

### 4.2 存储策略

```
用户发送 chat（第1轮） → 缓存，不存数据库
用户发送 chat（第2轮） → 缓存，不存数据库
用户发送 image       → 立即存数据库（chat 计数不变）
用户发送 chat（第3轮） → 缓存，不存数据库
用户发送 chat（第4轮） → 缓存，不存数据库
用户发送 chat（第5轮） → 批量存数据库，清空缓存，计数重置
```

### 4.3 流程图

```
┌─────────────────────────────────────────────────────────────────┐
│                        接收用户请求                               │
│                             ↓                                   │
│                    判断 modality 类型                            │
│                             ↓                                   │
│         ┌──────────────────┴──────────────────┐                │
│         ↓                                      ↓                │
│   modality == "chat"                    modality != "chat"      │
│         ↓                                      ↓                │
│   chat_count += 1                       立即存数据库             │
│   加入 pending_chats                    返回响应                 │
│         ↓                                              │
│   chat_count >= 5 ?                                            │
│         ↓                                                      │
│    ┌────┴────┐                                                 │
│    ↓         ↓                                                 │
│   是        否                                                  │
│    ↓         ↓                                                 │
│ 批量存储   返回响应                                              │
│ 清空缓存                                                       │
│ 返回响应                                                       │
└─────────────────────────────────────────────────────────────────┘
```

### 4.4 批量存储时的处理

当 chat_count 达到 5 轮时：

```
1. 将 5 轮对话合并为一个完整对话上下文
2. 调用 LLM 生成综合摘要（而非5次单独摘要）
3. 生成一个向量
4. 存入一条 Resource 记录（raw_content 包含完整对话）
5. 清空 pending_chats，重置 chat_count
```

**优势**：
- 减少 LLM API 调用（5 次 → 1 次）
- 减少数据库写入（5 次 → 1 次）
- 生成更连贯的对话摘要

## 6. 数据结构

### 6.1 Resource 表字段（对话摘要表）

| 字段 | 说明 |
|------|------|
| `id` | UUID 主键 |
| `user_id` | 用户 ID |
| `modality` | 模态（text/image/video/voice/document） |
| `raw_content` | **用户原始输入**（用户提问/图片路径/语音路径/视频路径） |
| `description` | **对话的综合摘要**（LLM 生成的客观第三人称描述，用于向量化） |
| `description_vector` | description 的 1536 维向量 |
| `importance_score` | 对综合摘要的整体重要性评分 (1-10)，详见 [打分细则](./IMPORTANCE_SCORING.md) |
| `assistant_response` | **AI 回复的总结**（LLM 对回答内容的精简摘要） |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间（合并/覆盖时更新，用于遗忘曲线计算） |

> **重要说明**：
> - Resource 表存储**对话粒度的综合摘要**，每条记录对应一次对话交互
> - `raw_content` 存储用户的原始提问/输入
> - `description` 存储 LLM 生成的对话综合摘要
> - `assistant_response` 存储 AI 回复的总结（而非完整回复）
> - 从对话中提取的原子化信息存储在 Category 表中，通过 ResourceCategory 关联表追踪来源

### 6.2 ResourceCategory 表字段（记忆来源关联表）

| 字段 | 说明 |
|------|------|
| `id` | UUID 主键 |
| `resource_id` | 对话摘要 ID |
| `category_id` | 原子化记忆 ID |
| `relation_type` | 关联类型：`created`/`updated` |
| `note` | 关联说明（可选） |
| `created_at` | 关联创建时间 |

> **关联类型说明**：
> | 类型 | 说明 |
> |------|------|
> | `created` | 该 Resource 首次创建了这个原子化记忆 |
> | `updated` | 该 Resource 更新了这个原子化记忆 |
>
> **重复提及的处理**：当用户重复提及某条记忆时，直接增加 `Category.importance_score`，不创建新的关联记录。

### 6.2 modality 处理说明

| modality | raw_content 存储 | description 来源 |
|----------|-------------------|-----------------|
| `text` | 原始文本 | LLM 生成摘要 |
| `image` | OSS 对象存储路径 | VLM/OCR 提取 |
| `video` | OSS 对象存储路径 | 帧抽取 + VLM |

## 8. 记忆生命周期管理

### 8.1 问题背景

当前系统只实现了记忆的"写入"操作，每次对话都会创建新的 Resource 记录。随着对话增多，会导致以下问题：

| 问题 | 影响 |
|------|------|
| **重复存储** | 用户多次提及同一信息，产生冗余记忆 |
| **信息过时** | 用户偏好改变，旧记忆未更新 |
| **记忆膨胀** | 数据库持续增长，检索效率下降 |
| **噪音干扰** | 低价值记忆淹没重要信息 |

### 8.2 生命周期概览

```
┌─────────────────────────────────────────────────────────────────────┐
│                        记忆生命周期                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   [新输入]                                                          │
│      │                                                              │
│      ▼                                                              │
│   ┌──────────────┐                                                 │
│   │ LLM 分类判断  │ → 确定目标分类                                   │
│   └──────────────┘                                                 │
│      │                                                              │
│      ▼                                                              │
│   ┌──────────────┐                                                 │
│   │ 分类内去重检查 │                                                 │
│   └──────────────┘                                                 │
│      │                                                              │
│      ├──────────────────────┬──────────────────────┐               │
│      ▼                      ▼                      ▼               │
│   重复                     相关                    无关             │
│   (≥0.90)               (0.75-0.90)              (<0.75)           │
│      │                      │                      │               │
│      ▼                      ▼                      ▼               │
│   ┌──────────┐         ┌──────────┐         ┌──────────┐          │
│   │ 强化已有  │         │ 合并/新建 │         │ 正常新建  │          │
│   └──────────┘         └──────────┘         └──────────┘          │
│      │                      │                      │               │
│      └──────────────────────┴──────────────────────┘               │

│   ┌──────────────────────────────────────────┐                     │

│   │  ┌─────────┐ ┌─────────┐ ┌─────────┐     │                     │

**关键流程说明**：

1. **LLM 先分类**：确定新输入应该归入哪个分类
2. **分类内去重**：只在该分类下检索相似记忆，避免跨分类误判
3. **三种处理路径**：
   - 重复 → 强化已有记忆（不新建）
   - 相关 → LLM 判断合并或新建
   - 无关 → 正常新建

### 8.3 去重机制（Deduplication）

#### 8.3.1 为什么在分类层级去重？

**问题：在 Resource 层去重效果差**

| 模态 | 去重时机 | 问题 |
|------|----------|------|
| Chat | 每 5 轮合并后存储 | 5 轮内容已变化，几乎不可能重复 |
| Image/Video/Voice | 立即存储 | 用户不会重复上传同一文件 |
| Document | 立即存储 | 用户不会重复上传同一文档 |

**真正需要去重的场景**：

| 场景 | 示例 | 去重价值 |
|------|------|----------|
| 重复表达偏好 | 第 1 天："我喜欢喝咖啡" / 第 10 天："我每天早上要喝咖啡" | 避免同一偏好存储多次 |
| 重复陈述事实 | 多次对话中提及"我是高中生" | 避免冗余身份信息 |
| 重复记录目标 | "我想考研" / "我的目标是考上研究生" | 避免目标碎片化 |

**结论**：去重应该在 **分类层级** 进行，即：LLM 确定目标分类后，在该分类下检查是否与已有记忆重复。

#### 8.3.2 去重流程

```

#### 8.3.3 相似度阈值设计

不同分类采用不同的去重阈值，以适应各分类的特性：

**Category 级别阈值**：

| 分类 | skip_threshold | merge_threshold | 说明 |
|------|----------------|-----------------|------|
| 核心自我 | 0.85 | 0.75 | 身份信息相对稳定，适度宽松 |
| 情景时间轴 | 0.90 | 0.75 | 情景细节需要更高相似度才跳过 |
| 语义知识库 | 0.88 | 0.75 | 知识信息需要较高的相似度 |
| 社交关系图谱 | 0.85 | 0.75 | 关系信息相对稳定 |
| 动态领域分类 | 0.92 | 0.75 | 动态分类需要非常高的相似度才跳过 |

**Resource 级别阈值**：

| 阈值类型 | 值 | 说明 |
|----------|-----|------|
| skip_threshold | 0.90 | Resource 是综合摘要，需要高相似度才跳过 |
| merge_threshold | 0.80 | 合并阈值相对较高 |

**默认阈值**（未定义分类）：

| 阈值类型 | 值 |
|----------|-----|
| skip_threshold | 0.88 |
| merge_threshold | 0.75 |

#### 8.3.4 不同分类的去重策略

| 分类 | 去重严格度 | 原因 |
|------|-----------|------|
| **核心自我** | 高（0.85） | 身份信息重复存储价值低 |
| **情景时间轴** | 中（0.90） | 时间点不同即视为不同事件 |
| **语义知识库** | 中（0.88） | 知识点可从不同角度记录 |
| **社交关系图谱** | 高（0.85） | 同一人信息应合并 |
| **动态领域分类** | 低（0.92） | 项目进度等需保留演进记录 |

#### 8.3.5 特殊场景处理

| 场景 | 检测方式 | 处理方式 |
|------|----------|----------|
| 用户补充细节 | 相似度 0.75-0.90 | 合并到已有记忆，追加新信息 |
| 用户修正信息 | LLM 检测矛盾 | 覆盖已有记忆，保留修正历史 |
| 用户重复强调 | 相似度 ≥ 0.90 | 跳过存储，强化已有记忆 |
| 信息演进 | 相似度 0.70-0.85 | 保留历史 + 新建记录 |

**补充细节示例**：

```
已有记忆："用户喜欢喝咖啡"
新输入："我每天早上喝美式，不加糖"

→ 检测到相似度 0.82
→ LLM 判断为补充信息
→ 合并结果："用户喜欢喝咖啡，每天早上喝美式，不加糖"
```

**修正信息示例**：

```
已有记忆："用户 23 岁，在字节跳动工作"
新输入："我其实 25 岁，之前说错了"

→ LLM 检测到矛盾
→ 覆盖结果："用户 25 岁，在字节跳动工作"
→ 可选：保留修正历史（"原记录：23岁，于2024-03-20修正"）
```

#### 8.3.6 实现要点

```python
async def check_duplicate_in_category(
    user_id: str,
    category_id: str,
    new_content: str,
    threshold: float = 0.90,
) -> dict:
    """在指定分类下检查新内容是否重复

    Args:
        user_id: 用户 ID
        category_id: 目标分类 ID
        new_content: 新内容摘要
        threshold: 重复阈值（默认 0.90）

    Returns:
        {
            "is_duplicate": bool,
            "is_related": bool,
            "similar_resource_id": str | None,
            "similarity_score": float,
            "action": "skip" | "merge" | "create"
        }
    """
    # 1. 生成新内容的向量
    new_vector = await llm.get_embedding(new_content)

    # 2. 获取该分类下所有 Resources
    resources = await resource_repo.get_by_category(category_id)

    if not resources:
        return {"is_duplicate": False, "is_related": False, "action": "create"}

    # 3. 计算与每条记忆的相似度
    max_similarity = 0.0
    most_similar_resource = None

    for resource in resources:
        similarity = cosine_similarity(new_vector, resource.description_vector)
        if similarity > max_similarity:
            max_similarity = similarity
            most_similar_resource = resource

    # 4. 根据相似度决定操作
    if max_similarity >= threshold:
        return {
            "is_duplicate": True,
            "is_related": True,
            "similar_resource_id": most_similar_resource.id,
            "similarity_score": max_similarity,
            "action": "skip",  # 跳过存储，后续强化已有记忆
        }
    elif max_similarity >= 0.75:
        return {
            "is_duplicate": False,
            "is_related": True,
            "similar_resource_id": most_similar_resource.id,
            "similarity_score": max_similarity,
            "action": "merge",  # 需要 LLM 进一步判断
        }
    else:
        return {
            "is_duplicate": False,
            "is_related": False,
            "similar_resource_id": None,
            "similarity_score": max_similarity,
            "action": "create",
        }
```

#### 8.3.7 强化已有记忆

当检测到重复时，不新建 Resource，而是强化已有记忆：

```python
async def reinforce_existing_memory(resource_id: str):
    """强化已有记忆（重复提及时触发）"""
    # 1. 增加强化计数
    await resource_repo.increment_reinforcement(resource_id)

    # 2. 小幅提升重要性（最多到 10）
    current_score = resource.importance_score
    new_score = min(current_score + 0.5, 10)
    await resource_repo.update(resource_id, importance_score=new_score)

    # 3. 更新最后访问时间（重置衰减计时）
    await resource_repo.update(resource_id, last_accessed_at=datetime.now())
```

### 8.4 更新机制（Update）

> **说明**：更新机制已整合到去重流程中。当检测到相关内容（相似度 0.75-0.90）时，由 LLM 判断是合并、更新还是新建。

#### 8.4.1 与去重流程的关系

```
分类内去重检查
      │
      ▼
相似度 0.75 - 0.90（相关内容）
      │
      ▼
┌─────────────────────────────────────────┐
│            LLM 判断操作类型              │
├─────────────────────────────────────────┤
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  │
│  │ 补充信息 │  │ 修正信息 │  │ 独立内容 │  │
│  └─────────┘  └─────────┘  └─────────┘  │
│       │            │            │       │
│       ▼            ▼            ▼       │
│   追加到已有    覆盖已有       新建      │
│   （合并）      （更新）       （独立）   │
└─────────────────────────────────────────┘
```

#### 8.4.2 三种操作类型

| 操作类型 | 检测条件 | 示例 |
|----------|----------|------|
| **合并（追加）** | 新信息补充已有记忆 | 已有："喜欢咖啡" → 新："每天早上喝美式" → 结果："喜欢咖啡，每天早上喝美式" |
| **更新（覆盖）** | 新信息修正已有记忆 | 已有："25岁" → 新："其实26岁了" → 结果："26岁" |
| **新建（独立）** | 主题相同但内容独立 | 已有："学Python" → 新："学机器学习" → 两条独立记录 |

#### 8.4.3 LLM 判断 Prompt

```
# Role
你是一个记忆管理专家。请判断新输入与已有记忆的关系。

# 已有记忆
{existing_memory}

# 新输入
{new_input}

# 任务
判断新输入与已有记忆的关系，并返回操作类型：
- "merge"：新信息是对已有记忆的补充，应合并
- "update"：新信息修正/覆盖已有记忆
- "create"：新信息是独立内容，应新建记录

# 输出格式
{
  "action": "merge" | "update" | "create",
  "reason": "判断理由",
  "merged_content": "合并/更新后的内容（仅 merge/update 时需要）"
}
```

#### 8.4.4 更新后的处理

| 操作 | 数据库处理 |
|------|-----------|

### 8.5 遗忘机制（Forgetting）

#### 8.5.1 设计目标

模拟人类记忆的遗忘曲线，自动降低不重要记忆的权重，最终删除。

**重要规则**：以下固定分类**不参与遗忘机制**，只增不改：
- **核心自我**：身份信息、偏好习惯是用户的核心特征，不应遗忘
- **社交关系图谱**：人际关系是长期记忆，不应遗忘

对于这两个分类，只支持：
- 新增信息（如新爱好、新朋友）
- 更新信息（如爱好变化、联系方式变更）

#### 8.5.2 遗忘曲线模型

采用 **艾宾浩斯遗忘曲线** 的改进版本，引入"访问加成"因子：

```
EffectiveScore = Importance × e^(-Days / (Importance × 5)) × (1 + log(AccessCount + 1))
```

**公式拆解**：
- `Importance`：原始重要性评分 (1-10)
- `Days`：距创建/最后访问的天数
- `AccessCount`：被检索引用的次数
- `e^(-Days / (Importance × 5))`：时间衰减因子（半衰期与重要性正相关）
- `(1 + log(AccessCount + 1))`：访问加成因子（对数增长，防止过度膨胀）

**修正后的衰减效果**：

| 原始分数 | 访问次数 | 30天后 | 90天后 | 180天后 |
|----------|----------|--------|--------|---------|
| 10 | 0 | 9.7 | 9.1 | 8.2 |
| 5 | 0 | 3.7 | 2.0 | 0.9 |
| 5 | 10 | 5.3 | 2.9 | 1.3 |
| 3 | 50 | 4.1 | 2.2 | 1.0 |

> **设计意图**：即使是随口提的小事，如果被 AI 反复引用，也会变成"长期记忆"。访问次数越多，记忆越难被遗忘。

#### 8.5.3 重要性衰减公式

```python
def calculate_decay(importance_score: int, days_since_access: int) -> float:
    """计算衰减后的有效重要性

    Args:
        importance_score: 原始重要性 (1-10)
        days_since_access: 距上次访问的天数

    Returns:
        有效重要性（可能小于原始值）
    """
    # 记忆强度系数：重要性越高，衰减越慢
    strength = importance_score * 10  # 10-100

    # 衰减率
    decay_rate = math.exp(-days_since_access / strength)

    # 有效重要性
    effective_importance = importance_score * decay_rate

    return effective_importance
```

#### 8.5.4 衰减示例

| 原始重要性 | 访问次数 | 30天后 | 90天后 | 180天后 |
|-----------|----------|--------|--------|---------|
| 10 (最高) | 0 | 9.7 | 9.1 | 8.2 |
| 5 | 0 | 3.7 | 2.0 | 0.9 |
| 5 | 10 | 5.3 | 2.9 | 1.3 |
| 3 | 50 | 4.1 | 2.2 | 1.0 |

> **设计意图**：访问次数可以显著延缓遗忘，即使是低重要性记忆，如果被反复引用，也会保留更久。

#### 8.5.5 遗忘触发条件

| 条件 | 操作 |
|------|------|
| 有效重要性 < 1 | 标记为"待遗忘" |
| 有效重要性 < 0.5 且超过 30 天未访问 | 彻底删除 |
| 用户主动删除 | 立即删除 |

#### 8.5.6 访问强化

记忆被检索或引用时，重置衰减计时：

### 8.6 整合机制（Consolidation）

#### 8.6.1 设计目标

#### 8.6.2 整合场景

### 8.7 时间轴清理机制

#### 8.7.1 设计目标

专门处理「情景时间轴」分类中的过时记忆。

#### 8.7.2 清理规则

| 记忆类型 | 时间条件 | 操作 |
|----------|----------|------|
| `happened`（已发生事件） | 超过 180 天 | 评估是否归档到「语义知识库」或删除 |
| `planning`（计划待办） | 时间已过且未标记完成 | 标记为过期或转为 `happened` |
| `planning`（计划待办） | 用户明确取消 | 删除 |

#### 8.7.3 过期待办处理

```python
async def cleanup_expired_todos(user_id: str):
    """清理过期的待办事项"""
    now = datetime.now()

    # 查找已过期但状态仍为 planning 的记忆
    expired_plannings = await resource_repo.find_expired_plannings(
        user_id=user_id,
        before=now,
    )

    for resource in expired_plannings:
        # 让 LLM 判断：已完成 / 已取消 / 仍需跟进
        status = await llm.check_todo_status(resource)

        if status == "completed":
            resource.timeline_context["status"] = "happened"
            await resource_repo.update(resource)
        elif status == "cancelled":
            await resource_repo.delete(resource.id)
        else:
            # 提醒用户跟进
            await send_reminder(user_id, resource)
```
